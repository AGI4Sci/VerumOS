# VerumOS · 设计文档 v2.1

> 本文档基于 design.md 原始愿景，结合架构讨论后的决策，作为重构指导文件。
> v2.1 新增：UI 是 Agent 定义的必要组成部分，Agent 上线 = 后端逻辑 + 前端 UI 同时就绪。

---

## 一、产品定位

**面向 AI 科学家和基础研究科学家的生命科学研究 Agent 平台。**

用户选择 Agent、描述任务，平台负责工具调用、代码生成和结果输出。每次执行自动生成完整的可复现代码包，用户可以在论文中引用、在任意环境中独立复现结果。

与普通 AI 工具的本质区别：**结构化交互 + 可复现输出 + 决策数据积累**，三者共同构成平台随时间加深的壁垒。

---

## 二、目标用户

### AI 科学家（ML 背景进入生命科学）
- 熟悉深度学习，不熟悉生命科学数据库和研究规范
- 痛点：不知道哪些方法适合特定问题，结果的生物学意义难以解读
- 需求：快速进入新的生命科学子领域，得到可信结果

### 基础研究科学家（生物 / 医学背景）
- 有明确科学问题，缺乏数据处理和建模的技术能力
- 痛点：依赖生信工程师，周期长，沟通成本高
- 需求：把科学想法直接变成分析结果，不管技术细节

两类用户对透明度需求不同：基础研究科学家默认隐藏执行细节；AI 科学家可以展开查看完整工具调用轨迹。**UI 默认隐藏，允许展开，不强制。**

---

## 三、核心设计原则

**原则一：结果可复现是硬约束。**
每次执行后自动生成可复现代码包，记录所有工具调用、参数和数据来源。科学家可以在论文方法部分完整描述流程，也可以在任意环境独立复现。

**原则二：不确定性隔离在生成阶段，确定性保证在执行阶段。**
LLM 生成代码的过程可以不稳定，但生成的脚本一旦通过沙箱验证，执行结果是确定的。这和 Jupyter Notebook 的使用模式一致：cell 怎么写的可以不确定，跑一遍 notebook 的结果是确定的。

**原则三：结构化 UI 是数据采集入口，不只是交互界面。**
固定的决策节点强制用户在同样的位置做选择，这些选择被记录下来，成为平台持续优化的原料。UI 的价值不只是好用，而是把非结构化的研究过程结构化，让平台能学习。

**原则四：先做稳，再做通用。**
三个 Agent 各自通用，能处理所属类别下的任意任务。但通用能力靠工具集的宽度支撑，不靠 LLM 猜测。没有工具支撑的能力不对外暴露。

**原则五：UI 是 Agent 的身体，不是平台的外壳。**
每个 Agent 自带一套 UI，UI 是 AgentDef 的组成部分，而不是所有 Agent 共用一个通用界面。Agent 的能力边界通过 UI 对用户可见——用户在 UI 上能看到什么输入项、什么决策节点、什么输出格式，就代表这个 Agent 能做什么。没有配套 UI 的 Agent 不算上线。

## 四、整体架构

```
┌─────────────────────────────────────────────────────────┐
│                        用户层                            │
│         选择 Agent → 描述任务 → 确认执行                  │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│                     Web UI 交互层                         │
│   预定义组件库：输入区 → 进度区 → 决策节点 → 输出区         │
│   默认隐藏执行细节，支持展开查看完整轨迹                    │
└───────────────────────────┬─────────────────────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│                    规则编排层（Phase 1）                   │
│   每个 Agent 有固定执行图，不做跨 Agent 动态编排            │
│   执行结束 → 自动生成可复现代码包 → 快照固化                │
└──────────────┬────────────┬────────────┬────────────────┘
               │            │            │
    ┌──────────▼──┐ ┌───────▼────┐ ┌────▼────────────┐
    │  Data Agent  │ │Model Agent │ │ Analysis Agent  │
    │ 数据处理     │ │ 代码生成   │ │  MCP 工具调用   │
    │ + 本地执行   │ │ + 沙箱验证 │ │  + 结果解读     │
    └──────────┬──┘ └───────┬────┘ └────┬────────────┘
               └────────────┴────────────┘
                            │
┌───────────────────────────▼─────────────────────────────┐
│                      Skill 库 / MCP 工具                  │
│   csv-skill / bioinfo-skill / MCP 计算工具 / 代码执行器    │
└─────────────────────────────────────────────────────────┘
```

### 关于编排的演进路径

**Phase 1（当前）：** 三个 Agent 各自独立，用户自己选 Agent，在 Agent 内部规则驱动执行。不做跨 Agent 编排。先把每个 Agent 单独跑稳。

**Phase 2：** 加入 Orchestrator，处理跨 Agent 任务。用户描述一个需要多个 Agent 协作的问题，Orchestrator 负责拆解和线性串联。规则硬编码，不做动态生成。

**Phase 3：** Orchestrator 成熟后，入口统一。用户只描述科学问题，系统自动决定激活哪些 Agent、以什么顺序执行。

**不要跳步。** Phase 1 没跑稳之前不启动 Phase 2。

## 五、三个核心 Agent

三个 Agent 的底层是同一套 AgentLoop + 声明式 AgentDef，差别只在工具集、system prompt 和 UI 布局声明。**这是架构最干净的地方，不要破坏它。**

### AgentDef 完整结构

```typescript
const agentDef = {
  id: 'data-agent',
  name: 'Data Agent',
  system_prompt: '...',
  tools: [...],
  routes: [...],
  memory_policy: {...},
  hooks: {...},
  ui_layout: {                        // ← UI 是 Agent 定义的一部分
    input: ['file-upload', 'task-input'],
    progress: ['progress-tracker'],
    decision: ['decision-card'],
    output: ['result-card', 'code-export']
  }
}
```

`ui_layout` 声明这个 Agent 在四区结构里各自使用哪些预定义组件。**AgentDef 不完整（缺少 ui_layout）的 Agent 不允许注册上线。**

---

### 5.1 Data Agent — 数据处理

**定位：** 通用数据处理 Agent，能处理生命科学数据处理类别下的任意任务。

**执行方式：** LLM + 工具调用 + 本地 Python 执行

**当前工具集：**
- `csv-skill`：CSV / TSV / Excel 读取、探索、转换、合并
- `bioinfo-skill`：单细胞表达矩阵处理、QC、标准化、marker 基因分析

**UI 布局：**
```
输入区：文件上传卡片 + 任务描述输入框
进度区：进度追踪面板（工具调用日志可展开）
决策区：数据质量报告 → 决策节点卡片（继续 / 重新处理 / 调整参数）
输出区：结果展示卡片 + 数据预览表格 + 代码导出按钮
```

**输入：** 原始数据文件 + 任务描述
**输出：** 标准化数据集 + 数据质量报告 + 数据处理代码

**扩展方向：** 接入 NCBI、UniProt、GEO 等数据库拉取工具，支持从数据库直接检索数据。

---

### 5.2 Model Agent — 模型构建

**定位：** 通用建模 Agent，核心能力是生成高质量的 AI 模型训练 / 推理代码，用户在本地（自有 GPU）执行。

**执行方式：** LLM 写代码 → 沙箱最小验证 → 交付用户

**沙箱验证机制（重要）：**
Model Agent 生成代码后，不直接交付，先在沙箱里用极小数据子集（如 100 条）跑 1 个 epoch，验证：
- 代码能无错运行
- loss 在正常范围
- 输出 shape 正确

验证通过才交付。这一步过滤大部分低级错误，用户拿到的代码置信度更高。

**当前工具集：**
- 代码生成（LLM 直接生成）
- 本地沙箱执行器（已有 Python 执行能力）

**UI 布局：**
```
输入区：参数配置面板（模型类型、超参数、数据路径）
进度区：进度追踪面板（代码生成 → 沙箱验证步骤可见）
决策区：沙箱验证报告 → 决策节点卡片（确认交付 / 调整后重新生成）
输出区：代码预览 + 验证报告展示 + 代码导出按钮
```

**输入：** 处理好的数据集 + 建模目标描述
**输出：** 模型训练 / 推理代码 + 沙箱验证报告 + 模型结果文件（用户本地跑后）

**扩展方向：** 封装 AlphaFold、ESMFold、scBERT 等领域预训练模型的调用接口作为工具。

---

### 5.3 Analysis Agent — 分析推理

**定位：** 通用分析 Agent，核心能力是调用 MCP 计算工具，对数据或模型结果进行深度分析，输出生物学解读和新假设。

**执行方式：** LLM + MCP 工具调用

**工具集（MCP 接入）：**
- 富集分析工具（GO、KEGG、Reactome）
- 网络分析工具（蛋白质互作网络、基因调控网络）
- 文献检索工具
- 数据库查询工具

MCP 工具接入是标准化的，新工具按 MCP 协议接入即可，不需要修改 Agent 逻辑。**这是 Analysis Agent 可扩展性的核心。**

**UI 布局：**
```
输入区：文件上传卡片（数据或模型结果）+ 分析目标输入框
进度区：进度追踪面板（MCP 工具调用序列可展开）
决策区：假设列表卡片（多个假设 + 证据强度，用户选择深入方向）
输出区：结果展示卡片 + 可视化容器 + 分析代码导出 + 报告导出按钮
```

**输入：** 数据集或模型结果 + 分析目标
**输出：** 分析结果 + 生物学解读 + 新假设列表 + 分析代码 + 可视化图表

## 六、Web UI 设计

### 6.1 UI 存在的三个理由（按优先级排序）

1. **积累决策数据：** 固定的决策节点让用户的选择可被记录，这是平台最核心的长期资产
2. **固化研究范式：** 结构化交互让研究轨迹可复现、可协作、可复盘
3. **承载信息密度：** 科研任务的信息密度远超普通对话，表单、进度追踪、图表展示是必要的

### 6.2 整体交互流程

```
① Agent 选择
   用户选择 Data / Model / Analysis Agent
   如果任务描述不清晰，进入多轮澄清对话
   任务确认后进入执行页面
         ↓
② 执行过程（统一四区结构）
   输入区：结构化表单 / 文件上传 / 参数配置
   进度区：当前步骤 + 工具调用日志（默认折叠，可展开）
   决策区：关键节点弹出，用户选择方向（非关键节点自动跳过）
   输出区：结果展示 + 可视化 + 代码下载 + 下一步引导
         ↓
③ 结果导出
   一键导出可复现代码包
   一键导出结构化分析报告
```

### 6.3 标准页面四区结构

每个 Agent 的交互页面使用统一结构，具体内容不同：

```
┌──────────────────────────────────────────┐
│  输入区                                   │
│  结构化表单 / 文件上传 / 参数配置           │
├──────────────────────────────────────────┤
│  进度区                                   │
│  执行步骤列表 + 工具调用日志（默认折叠）    │
│  [展开] 查看完整执行轨迹                   │
├──────────────────────────────────────────┤
│  决策区（仅在关键判断节点出现）             │
│  系统生成选项 + 支持证据，用户选择方向      │
│  用户可选择"继续"或"调整后重跑"            │
├──────────────────────────────────────────┤
│  输出区                                   │
│  结果展示 / 可视化 / 代码下载 / 下一步     │
└──────────────────────────────────────────┘
```

### 6.4 预定义 UI 组件库

**核心原则：不用 AI 实时生成 UI，预定义组件集合，AI 负责从中组装。**

第一批优先实现的组件（覆盖三个 Agent 的核心交互）：

| 组件 | 用途 | 优先级 |
|------|------|--------|
| 进度追踪面板 | 步骤列表 + 当前状态，可折叠展开工具调用日志 | P0 |
| 决策节点卡片 | 展示多个选项 + 支持证据，用户点击选择 | P0 |
| 结果展示卡片 | 结论 + 置信度 + 来源溯源 | P0 |
| 文件上传卡片 | 支持 csv、h5ad、fasta 等格式，上传后预览 | P1 |
| 参数配置面板 | 带默认值和说明的参数输入，用于模型超参等 | P1 |
| 可视化容器 | 通用图表嵌入（热图、UMAP、网络图等） | P1 |
| 代码导出按钮 | 触发可复现代码包打包下载 | P1 |
| 假设列表卡片 | 展示多个假设及证据强度，支持筛选标注 | P2 |
| 报告导出按钮 | 下载结构化分析报告 | P2 |

P0 组件先做，P2 组件后补。**不要同时开发全部组件。**

## 七、可复现代码包（核心特性）

**这是平台区别于所有普通 AI 工具的最重要特性，也是学术增长飞轮的起点。**

每次 Agent 执行结束后，自动生成标准结构的代码包：

```
research_session_[jobId]/
├── README.md           # 研究问题、执行摘要、主要发现
├── data_pipeline.py    # 数据处理完整代码（Data Agent 生成）
├── model_pipeline.py   # 模型训练 / 推理代码（Model Agent 生成）
├── analysis.py         # 分析步骤代码（Analysis Agent 生成）
├── run_all.py          # 一键复现入口
├── requirements.txt    # 依赖包版本锁定
├── params.json         # 所有参数记录
└── data/
    └── checksums.md    # 数据来源 URL + MD5 校验值
```

**交付标准：** 任何人拿到这个文件夹，执行 `python run_all.py`，得到相同结果。

**实现节奏：**
- Phase 1：把当前生成的脚本 + job.json 参数打包成 zip，加 README，先跑通导出流程
- Phase 2：补充 requirements.txt 自动生成、数据 checksum 记录
- Phase 3：run_all.py 自动编排多个 pipeline 的执行顺序

**不要等功能完善再做这个，第一版哪怕只是打 zip，也要作为独立功能暴露出来。** 它传递给用户的信号比功能本身更重要：你的结果是可信的、可引用的。

## 八、决策数据记录机制（核心壁垒）

**这是平台最不可复制的长期资产。** 通用 AI 工具可以复制功能，但复制不了科学家在生命科学任务上的真实决策数据。

### 什么是决策节点

执行过程中需要用户主动判断的节点，例如：
- 数据质量报告出来后：数据是否符合预期？继续还是重新处理？
- 模型评估结果出来后：指标是否满意？继续还是调整参数重跑？
- 分析假设生成后：选择哪个假设方向深入？

### 记录结构

在现有 `job.json` 的 `traces` 基础上，增加决策事件类型：

```json
{
  "type": "decision_point",
  "timestamp": "...",
  "context": {
    "agent": "data-agent",
    "stage": "data_quality_review",
    "system_recommendation": "数据质量良好，建议继续",
    "options_presented": ["继续", "重新处理", "调整参数"]
  },
  "user_choice": "继续",
  "time_to_decide_seconds": 12
}
```

**现在就要把这个结构定义好，** 哪怕 Phase 1 只记录不分析。数据结构定义晚了，补录成本极高。

### 长期用途

- 分析哪些决策路径最终产出了高质量结果
- 优化 Agent 的默认推荐和参数选择
- 积累到足够规模后，训练生命科学专属的 Agent 推荐模型

## 九、Agent 扩展机制

**目标：新 Agent 从需求到上线 1-3 天，边际成本随 Agent 数量增加而下降。**

这个目标能实现的前提是基础设施标准化：UI 组件可复用，Skill / MCP 工具可复用，AgentDef 格式统一。

### UI 组件库的定位

**UI 组件库是 Agent 上线的基础设施，与 SkillRegistry、ToolRegistry 同等地位。**

三者的关系：
- `SkillRegistry / ToolRegistry`：Agent 的"能力"从哪里来
- `UIComponentLibrary`：Agent 的"身体"从哪里来
- `AgentDef`：声明这个 Agent 用什么能力、长什么样

组件库越丰富，新 Agent 上线越快。组件库不够，新 Agent 就卡在"没有合适的 UI 来承载这个交互"上。**每次新增组件都是对所有未来 Agent 的投资。**

### Agent 上线标准（v2.1 更新）

一个 Agent 上线必须同时满足：

| 条件 | 说明 |
|------|------|
| ✅ 后端可执行 | AgentDef 注册完整，工具链端到端跑通 |
| ✅ 前端 UI 完整 | ui_layout 声明的所有组件均已实现并联调 |
| ✅ 决策节点已定义 | 至少一个决策节点配置了 decision_point 记录 |
| ✅ 沙箱验证通过 | 用测试数据跑完整流程，输出符合预期 |

**缺少任何一条，Agent 不允许上线。**

### 扩展流程

```
① 用户 / 内部提出新 Agent 需求
② 评估现有 UI 组件是否覆盖
     ├── 能覆盖 → 直接进入 ③
     └── 不能覆盖 → 先开发缺失组件，加入组件库，再进入 ③
③ 评估需要接入哪些新工具（Skill 或 MCP）
④ 编写完整 AgentDef（system prompt + tools + ui_layout）
⑤ 后端沙箱验证 + 前端联调，确认四区结构完整可用
⑥ 上线，进入 Agent 选择池
```

### 标准化要求

**AgentDef 格式必须统一。** 新 Agent 只声明 system prompt、tools、routes、memory policy、ui_layout，不允许直接修改 AgentLoop。所有个性化行为通过 hooks 表达。

**工具接入必须走注册表。** 新 Skill 按 SKILL.md 格式描述，新 MCP 工具按 MCP 协议接入，两者都通过 ToolRegistry / SkillRegistry 统一管理，不允许绕过。

**UI 组件必须来自组件库。** 不允许在 Agent 内部直接写一次性 UI 代码。需要新交互形式时，先把组件加入组件库，再在 AgentDef 里引用。

## 十、核心壁垒

壁垒来自两层积累，随时间自动加深，不需要主动维护：

### 第一层：研究行为数据

科学家在决策节点的真实选择数据，是平台专属的、不可复制的训练数据。积累到足够规模后可以：
- 让 Agent 推荐越来越准（比用户自己选更好）
- 形成"研究范式库"，新任务直接复用验证过的路径

### 第二层：可复现代码的引用网络

科学家在论文中引用 VerumOS 生成的 pipeline，形成学术可见度。其他科学家发现后成为新用户，构成增长飞轮。这个飞轮一旦启动，普通 AI 工具因为没有可引用的标准输出，无法进入这个循环。

## 十一、实现优先级与节奏

### 现在立刻做（Phase 1 重构目标）

**① 打磨 Data Agent 到足够稳定**
- 它是 Model Agent 和 Analysis Agent 的上游，不稳则全局抖
- 核心指标：任意 csv / h5ad 输入，生成的处理脚本沙箱验证通过率 > 90%

**② 实现可复现代码包导出（第一版）**
- 把现有生成的脚本 + params.json 打包 zip，加 README
- 加一个导出 API 和 UI 里的导出按钮
- 哪怕第一版很简陋，这个功能要在 Phase 1 就上线

**③ 定义决策数据记录结构**
- 在 job.json 里增加 decision_point 事件类型
- Phase 1 只记录不分析，但结构必须现在定义

**④ 实现 P0 UI 组件（进度追踪、决策节点卡片、结果展示卡片）**
- 这三个覆盖了所有 Agent 的核心交互节点
- 其余组件后续迭代补充

### 接下来做（Phase 2）

**⑤ 实现 Analysis Agent**
- MCP 工具调用是纯增量接入，架构风险最低
- 先接入 2-3 个高频工具（富集分析、文献检索）验证 MCP 接入路径

**⑥ 完善可复现代码包**
- 补充 requirements.txt 自动生成
- 补充数据 checksum 记录
- 实现 run_all.py 自动编排

### 最后做（Phase 3）

**⑦ 实现 Model Agent**
- 代码生成 + 沙箱最小验证是最需要打磨的组合
- 放最后，有前两个 Agent 的经验积累再做更稳

**⑧ 实现 Orchestrator（跨 Agent 编排）**
- 等三个 Agent 各自稳定后，才有条件抽取编排规则
- 先硬编码 3 个场景的跨 Agent 路径，再考虑抽象成配置

## 十二、明确不做的事（边界）

- **不做 LLM 实时生成 UI**：稳健性不足，所有交互走预定义组件
- **不做平台侧 GPU 计算资源**：Model Agent 生成代码，用户本地执行
- **Phase 1 不做跨 Agent 动态编排**：先让三个 Agent 各自稳定
- **暂不建领域知识层**：先用通用 LLM 能力覆盖，等有领域合作者再补
- **不做 AI 猜测用户意图的路由**：Phase 1 用户主动选 Agent，减少误判
