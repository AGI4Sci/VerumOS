AI-Native 生命科学对外路线图

不是再做一个会调工具的 biomedical agent，而是建设可复用、可沉积、可迁移的科研决策基础设施

核心定位：我们不只自动化分析流程；我们把 专家纠偏、知识时效 与 阴性结果 变成可复用的数据资产，持续提升生命科学研究中的判断质量、协作效率与决策可靠性。

对外叙事不应停留在 “Skills 更多、工具更全、自动化更强”。真正的差异在于：别人主要帮助用户更快得到结论，我们要帮助用户更可靠地判断 哪个结论值得相信、为什么值得相信、哪些路径不该再重复尝试。

为什么现在做

工具调用型 agent 已经普及，但生命科学真正稀缺的是可追溯判断、跨机构经验沉积与对失败路径的系统管理。

我们卖什么

不是一次性分析报告，而是不断增值的研究对象：证据、推理、纠偏、争议与负路径一起沉积。

最终护城河

随着真实使用增长，系统积累的不是日志，而是别人拿不到的领域判断资产，可进一步蒸馏为高自主度 agent 与领域模型。

阶段一设置了明确的「通关条件」：如果没有形成真实产品价值与可沉积资产，就不进入下一阶段。目标不是堆功能，而是先形成无法轻易复制的科研决策闭环。点击展开详情。

阶段一 · 可信科研副驾驶 Demo

0–6个月 · 产品：公开数据上的高价值场景闭环 + 三类核心资产的数据结构

现在启动 ▶

这一阶段必须回答的问题

我们能否先不用强调“全自动科研”，而是先做出一个用户愿意拿来讨论真实问题的 可信科研副驾驶：它不仅给结论，还能告诉用户证据来自哪里、哪些地方存在冲突、哪些常见路径已经被排除，以及专家为什么会修改 AI 的判断。

阶段一对外最重要的认知，不是“我们也有 biomedical skills”，而是：这是第一个把专家纠偏、知识时效、阴性结果同时产品化的生命科学科研副驾驶。
💬 对外一句话定位：其他 AI 告诉你答案是什么，我们告诉你这个答案值不值得赌上你的实验经费。

对外 demo 应该长什么样

不要做：一个“给我数据，我自动出报告”的通用分析 demo。这种形态很容易被理解为又一个基于 skills 的 biomedical agent 壳层。

要做：一个围绕高价值科研问题的闭环 demo，输出至少包含五个模块：当前建议、支持证据、反对证据、时效/争议状态、已知无效路径、专家修正版判断。

最适合的首个场景：“文献证据链 + 靶点/通路优先级建议” 优先于 “单细胞自动注释”。前者更能体现证据冲突、时间更新、负路径排除与专家纠偏的价值。

阶段一要沉积的不是 skills 数量，而是可复用资产结构

资产 1｜专家纠偏轨迹

AI 初始判断、专家修改内容、修改理由、引用依据、替代结论。核心价值是把“专家不同意 AI 的瞬间”变成高质量训练数据。

资产 2｜有时间轴的知识节点

每条知识都记录时间戳、证据强度、争议状态与来源。系统不会把新旧结论静默覆盖，而会显式呈现冲突与更新。

资产 3｜阴性结果 / 无效路径库

把“这条路试过但没有用”作为一等对象管理。长期看，这比正向案例更能节省科研成本，也更容易形成跨机构网络效应。

需要构建的基础能力

执行层：把专家流程结构化为可执行 protocol，但对外不要把它作为主要卖点；它是基础设施，不是品牌心智。

呈现层：所有输出必须自带证据来源、判断依据、争议标记与已排除路径，确保结果可辩护、可讨论、可复查。

交互层：在关键步骤预设专家介入点，让修正过程被记录，而不是只保留最终结论。

对外差异化表达

用户为什么不用 Biomni / OpenClaw 类产品，而选择我们：它们已经证明了 tool orchestration 和 skill loading 可以规模化；例如 Biomni 将自己定位为通用 biomedical AI agent，OpenClaw 则以 SKILL.md 机制加载并筛选技能。 但对于真实科研协作，用户更缺的不是“更多工具”，而是 更可信的判断结构、更可追溯的专家分歧、以及对失败路径的系统管理。

通关条件

完成 1 个高价值 demo 闭环，且至少有 1 位外部合作方愿意在真实讨论中使用；输出中 专家纠偏、知识时效、阴性结果 三类结构全部上线；系统能稳定生成“建议 + 证据 + 反证 + 时效 + 负路径 + 专家修正”的标准对象；用户反馈的核心价值点是“更可信、更可辩护”，而不是只评价“跑得更快”。

阶段一基础服务：Data Agent 与 Model Agent

Demo 的最小可交付闭环由两个 Agent 支撑。Data Agent 负责将原始生物数据转化为可溯源的结构化数据对象；Model Agent 接收 Data Object 与建模需求，执行分析并交付带推理链的高质量模型结果。两者串联运行，输出共同写入 Shared Asset Store，为三类核心资产的沉积打地基。

Data Agent · 职责与模块

Collector：对接公开数据库（GEO、PubMed、TCGA、UniProt），支持用户上传私有数据（RNA-seq、scRNA-seq、临床表型）。

Cleaner：处理缺失值、批次效应与格式不一致；对文献数据提取结构化字段（PMID、摘要、结论、发表时间）。

Normalizer：统一 ontology 标注（HPO 表型、GO 功能注释、MONDO 疾病分类）；基因 ID 归一化（Ensembl / HGNC）。

输出 Data Object：source · created_at · quality_score · processing_history · ontology_tags。所有字段必须填写，不允许静默缺失。

Model Agent · 职责与模块

Protocol Selector：根据数据类型与用户需求匹配分析模板（差异表达、生存分析、靶点优先级排序等）；记录模板选择理由。

Model Runner：执行分析；记录超参数、模型选择依据与关键中间结果；核心参数（p_value_threshold、fold_change_cutoff 等）可由用户配置。

输出 Model Object：result · reasoning_trace · model_params · excluded_paths · protocol_version · created_at。推理链与排除路径为必填字段，直接写入 Asset Store 供后续阶段复用。

协作约定

Data Object 是两个 Agent 之间的接口契约，schema 在开发启动前锁定，不得单边修改。Model Agent 只消费经过 Normalizer 处理的 Data Object，不直接读取原始文件。两者输出均写入 Shared Asset Store（专家纠偏轨迹、知识时效节点、阴性结果库）。数据分析 Agent 在 Data Agent 与 Model Agent 稳定交付后再启动，不并行开发。
