# AI + 生命科学研究 Agent 平台设计文档

---

## 一、产品定位

**面向 AI 科学家和基础研究科学家的生命科学研究 agent 平台。**

用户输入研究问题，平台自动编排 workflow，调用工具和数据库，输出可重复、可溯源的研究结果。用户全程看不到内部 workflow 的编排细节，但每次执行都会生成完整的代码，确保结果可以被独立复现。

---

## 二、目标用户

### AI 科学家（ML 背景进入生命科学）
- 熟悉深度学习和数据分析，但不熟悉生命科学领域数据库和研究规范
- 痛点：不知道哪些数据源权威、哪些方法适合特定问题、结果的生物学意义如何解读
- 需求：快速进入一个新的生命科学子领域，得到可信的结果

### 基础研究科学家（生物/医学背景）
- 有明确的科学问题，但缺乏数据处理和模型训练的技术能力
- 痛点：依赖生信工程师合作，周期长，沟通成本高
- 需求：把科学想法直接变成分析结果，不需要管技术细节

---

## 三、核心设计理念

> **对用户暴露研究场景，对系统暴露执行能力。**

用户的交互语言是科学问题，系统的执行语言是工具调用和代码。平台负责在两者之间做翻译，用户不需要理解中间过程。

**结果可复现是硬约束。** 每次 workflow 执行后，系统自动生成完整的可运行代码（Python/R），记录所有工具调用、参数选择和数据来源，确保科学家可以在论文方法部分描述完整流程，也可以在任何环境中独立复现。

## 五、三类核心 Agent

Agent 的切分遵循 AI 科学家的工作视角：数据准备好，模型工具准备好，然后做分析推理。三类 agent 不是孤立的模块，而是一个研究流程的三个阶段，可以线性串联，也可以迭代循环。

### 5.1 Data Agent — 数据处理

**职责：** 把原始的、分散的、格式不一的生命科学数据，变成可以直接用于模型训练或分析的标准化数据集。

**核心能力：**
- 多数据库整合：从 NCBI、UniProt、GEO、ChEMBL、PDB 等主流数据库拉取数据，统一格式
- 数据清洗：处理缺失值、重复项、批次效应（针对组学数据的特殊处理）
- 特征工程：根据下游任务类型，自动选择合适的特征表示方式
- 数据质量报告：输出数据质量评估，标注可能影响结果的数据问题

**领域知识层（由领域合作者维护）：**
- 哪些数据库对哪类问题权威
- 各类组学数据的标准预处理流程（如 scRNA-seq 的质控标准）
- 常见批次效应的来源和处理方法

**输入：** 研究问题描述 + 原始数据文件或数据库检索条件

**输出：** 标准化数据集 + 数据质量报告 + 数据处理代码

---

### 5.2 Model Agent — 模型构建

**职责：** 根据研究问题和数据特点，选择、配置并运行合适的模型，输出可解释的模型结果。

**核心能力：**
- 模型选择推荐：给定问题类型和数据规模，推荐并解释为什么选择某个模型
- 预训练模型调用：封装 AlphaFold、ESMFold、scBERT、MolBERT 等领域专用模型的调用接口
- 自定义模型训练：支持用户提供自己的模型架构，平台负责训练流程管理
- 模型评估：输出领域相关的评估指标，而不只是通用的 loss/accuracy

**领域知识层：**
- 哪些预训练模型适合哪类生命科学任务
- 领域特定的评估标准（如蛋白质结构预测中的 TM-score vs RMSD）
- 模型结果的生物学合理性检查

**输入：** 标准化数据集 + 问题类型（分类、回归、生成、结构预测等）

**输出：** 模型结果文件 + 评估报告 + 训练/推理代码

---

### 5.3 Analysis Agent — 分析推理

**职责：** 调用工具和数据库，对模型结果或原始数据进行深度分析，发现新的生物学知识。

**核心能力：**
- 富集分析：GO、KEGG、Reactome 通路分析
- 网络分析：蛋白质互作网络、基因调控网络构建与分析
- 文献关联：把分析结果与已发表文献自动关联，标注哪些是已知发现，哪些可能是新发现
- 假设生成：基于分析结果，生成可检验的生物学假设，并评估每个假设的支持证据强度
- 可视化：根据数据类型自动选择合适的图表（热图、UMAP、网络图等）

**领域知识层：**
- 各类分析方法的适用条件和局限性
- 结果显著性的生物学意义解读（p-value 在生物学语境下的理解）
- 什么样的发现值得进一步实验验证

**输入：** 模型结果或数据集 + 分析目标

**输出：** 分析结果 + 生物学解读 + 新假设列表 + 分析代码 + 可视化图表

## 六、Web UI 设计

### 6.1 Web UI 存在的三个理由

1. **固化研究范式：** 把非结构化的研究过程结构化，让研究轨迹有章可循，便于团队协作和研究复盘
2. **积累行为数据：** 记录科学家在每个决策节点的选择，沉淀为平台的核心数据资产，持续优化 workflow
3. **结构化交互：** 科研任务的信息密度远高于普通对话，需要表单、文件上传、进度追踪、图表展示等结构化组件，纯聊天界面无法承载

### 6.2 整体交互流程

```
① 问题入口
   用户用自然语言描述研究问题
   系统解析问题，推荐应激活哪些 agent 及执行顺序
   用户确认或调整
         ↓
② 数据准备（Data Agent 页面）
   用户上传数据或填写数据库检索条件
   系统执行数据处理，进度实时可见
   [决策节点] 数据质量报告展示，用户确认数据是否符合预期
         ↓
③ 模型运行（Model Agent 页面）
   系统推荐模型，用户可调整关键参数
   模型训练/推理在后台运行，进度可见
   [决策节点] 模型评估结果展示，用户判断是否继续
         ↓
④ 分析推理（Analysis Agent 页面）
   系统自动执行分析，生成假设列表
   [决策节点] 用户从多个假设中选择感兴趣的方向
         ↓
⑤ 结果输出
   完整结果展示：结论 + 置信度 + 来源溯源
   一键导出：分析报告 + 完整可复现代码
```

### 6.3 标准页面结构

每个 agent 的交互页面使用统一的四区结构，只有具体内容不同：

```
┌──────────────────────────────────────┐
│  输入区                               │
│  结构化表单 / 文件上传 / 参数配置       │
├──────────────────────────────────────┤
│  进度区                               │
│  当前执行步骤 / 工具调用日志（可折叠）  │
├──────────────────────────────────────┤
│  决策区（仅在需要人工判断时出现）        │
│  系统生成多个选项，用户选择方向          │
├──────────────────────────────────────┤
│  输出区                               │
│  结果展示 / 可视化 / 代码下载 / 下一步  │
└──────────────────────────────────────┘
```

### 6.4 预定义 UI 组件库

不使用 AI 实时生成 UI（稳健性不足），而是预定义一套组件，AI 负责从中组装：

| 组件类型 | 用途 |
|---|---|
| 文件上传卡片 | 原始数据上传，支持常见格式（csv、h5ad、fasta 等） |
| 数据库检索表单 | 配置数据库查询条件，带字段说明 |
| 参数配置面板 | 模型超参数、分析阈值等，带默认值和解释 |
| 进度追踪面板 | 步骤列表 + 当前状态 + 预计时间 |
| 假设选择卡片 | 展示多个假设及支持证据，支持筛选和标注 |
| 结果展示卡片 | 结论 + 置信度 + 引用来源 |
| 可视化容器 | 嵌入热图、UMAP、网络图等 |
| 代码导出按钮 | 下载完整可复现代码 |
| 报告导出按钮 | 下载结构化分析报告 |

## 七、可复现代码机制

**这是平台区别于普通 AI 工具的核心特性。**

每次 workflow 执行结束后，系统自动生成：

```
research_session_[timestamp]/
├── README.md          # 研究问题、执行摘要、主要发现
├── data_pipeline.py   # 数据下载、清洗、预处理的完整代码
├── model_pipeline.py  # 模型配置、训练/推理代码
├── analysis.py        # 分析步骤、工具调用代码
├── requirements.txt   # 依赖包版本
├── params.json        # 所有参数记录
└── data/
    └── checksums.md   # 数据来源 URL 和 MD5 校验值
```

任何人拿到这个文件夹，在任何环境中执行 `python run_all.py`，都应该得到相同的结果。这是发表论文的方法可重复性要求，也是科学家使用平台的信任基础。

## 八、workflow 文档机制

每个 agent 的执行逻辑用 `.md` 文档描述，包含：
- 执行目标
- 输入输出格式
- 工具调用顺序和条件判断
- 领域知识注释（为什么选这个工具、这个参数）

**第一次执行：** 系统根据问题动态生成 workflow 文档

**执行后：** 文档固化为版本快照，与生成的代码一起存储

**下次类似问题：** 直接复用快照，减少等待时间，提高结果稳定性

随着积累，高质量的 workflow 快照形成**研究范式库**，这是平台最核心的长期资产。

## 九、新 Agent 扩展路径

平台上线后，用户可以提交新的 agent 需求：

```
用户描述研究需求
      ↓
系统从 UI 组件库自动组装交互页面
      ↓
领域合作者审核并编写 .md workflow 文档
      ↓
系统验证 workflow 可执行性
      ↓
新 agent 上线，进入 agent 池子
```

**目标：** 新 agent 从需求提交到上线的周期控制在 1-3 天，而不是"几天时间自行构建"。边际成本随 agent 数量增加而下降，因为 UI 组件和底层 skill 可以复用。

## 十、核心壁垒

平台的壁垒来自两层积累，随时间自动加深：

**领域知识层（由合作者共建）**
- 每个 agent 背后的数据规范、评估标准、结果解读逻辑
- 这是通用 AI 工具无法替代的，是平台可信度的来源

**研究行为数据层（由用户使用积累）**
- 科学家在哪些决策节点犹豫、选择了哪条路径、最终结果如何
- 用于持续优化 workflow，训练更好的 agent 推荐模型
- 积累到一定规模后，可以训练专属于生命科学研究的 agent 模型
